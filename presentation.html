Tool-Agnostic AI Definitions
How to write Rules, Agents & Skills that work everywhere
📜 Rules 🤖 Agents ⚡ Skills
Context
Why Tool-Agnostic?
Instructions travel across AI tools — write them once, apply everywhere.
🔄
Portability
Currently supported in Claude Code and GitHub Copilot. The concepts are designed to extend to future tools as the ecosystem matures.
🧩
Composability
Rules, agents, and skills compose together. A skill may invoke agents; agents follow rules.
📐
Consistency
Teams get predictable AI behavior across every developer's local setup and CI pipeline.
Overview
The Three Primitives
A quick reference before we dive in.
📜
Rules / Instructions
Persistent behavioral guidelines. They are always active and shape how AI responds in every context.
🤖
Agents
Specialized, autonomous task-runners with a defined role, tool access, and operating scope.
⚡
Skills
Reusable, invocable procedures that encode a specific workflow step or repeatable capability.
Architecture
The Hierarchy
Each layer builds on the one above it.
Layer 1 — Always Active
📜 Rules / Instructions
Project conventions, tone, safety constraints, coding standards
↓ consumed by
Layer 2 — On Demand
🤖 Agents
Scoped task-runners that operate within the rules and use available tools
↓ invoked by
Layer 3 — Reusable Steps
⚡ Skills
Atomic procedures that agents (or users directly) call to accomplish sub-tasks
Layer 1
Rules & Instructions
The always-on behavioral contract for AI in your project.
🗂️
Where they live
CLAUDE.md (Claude Code), copilot-instructions.md (GitHub Copilot) — the instruction file your tool loads on start-up.
⏱️
Activation
Passively applied to every AI interaction without needing to be invoked. They are always in scope.
🌐
Scope
Can be global (whole repo), layer-scoped (frontend / backend), or file-type-scoped. Compose via @include or similar directives.
Rules
What to Include — and Avoid
✅ Include
  • Coding conventions (naming, formatting, patterns)
  • Response tone and greeting format requirements
  • Security constraints (never commit secrets, avoid eval)
  • What the AI should NOT do by default
  • Language/framework preferences
  • Prohibited patterns (e.g. no raw SQL in controllers)
❌ Avoid
  • Step-by-step workflow logic (→ Skills)
  • Tool-specific API calls or CLI flags
  • One-shot task descriptions (→ Agents)
  • Dynamic context or file lists (→ agent prompt)
  • Information derivable from code/git history
  • Ephemeral state or in-progress task notes
  • Duplicate info already in config files
Rules · Example
Anatomy of Good Rules
# General guidelines (CLAUDE.md / copilot-instructions.md)

## Tone
- Begin every response with the 👋 emoji.
- Keep answers concise; prefer code over prose.

## Code style
- Use camelCase for variables, PascalCase for types.
- Never add speculative error handling for impossible cases.
- Prefer editing existing files over creating new ones.

## Security
- Do not generate or commit secrets, tokens, or credentials.
- Validate all user input at system boundaries.
Tip: Keep a single root file and compose layer-specific overrides via includes — this prevents duplication and makes the hierarchy explicit.
Layer 2
Agents
Scoped AI workers with a single well-defined role.
🎯
Purpose
Each agent encapsulates a role: "code reviewer", "test runner", "migration planner". One agent, one job.
🛠️
Tools
Declare only the tools the agent needs. Minimal tool access reduces surface area and keeps the agent focused.
📦
Isolation
Agents run in their own context but always operate within global rules — rules have higher priority and cannot be overridden by an agent.
Agents
What to Include — and Avoid
✅ Include
  • Clear, single-sentence role description
  • Explicit list of allowed tools (whitelist)
  • Output format expectations (report, diff, JSON…)
  • Scope boundaries (which files/dirs to touch)
  • When to stop and ask vs. proceed autonomously
  • When NOT to use this agent (usage guidance)
  • Model override if the task warrants it
❌ Avoid
  • Mixing multiple unrelated responsibilities
  • Duplicating global rules already in instructions
  • Hardcoded file paths or branch names
  • Granting tools the agent will never use
  • Procedural step-by-step instructions (→ Skills)
  • Storing task-state or in-progress work inside the agent
  • Tool-specific syntax (e.g. Claude-only flags)
Agents · Example
Anatomy of a Well-Defined Agent
---
name: code-reviewer
description: Reviews pull-request diffs for correctness,
             security, and style. Use for PR review tasks.
model: sonnet                     # optional override
tools:
  - Read                         # can read files
  - Grep                         # can search code
  - Bash                         # can run git diff
  # no Write, no Edit — read-only role
---

# Role
You are a senior code reviewer. Your job is to identify bugs,
security issues, and style violations in the provided diff.

# Output
Return a bullet-point report. Group findings by severity:
Critical → Warning → Suggestion. Under 300 words.
Layer 3
Skills
Reusable, invocable procedures — the verbs of your AI toolkit.
🔁
Reusability
A skill encodes a workflow once. Any agent or user can invoke it via /skill-name or an explicit reference.
🔬
Atomicity
Each skill does one thing well. Complex workflows chain skills; skills themselves stay small and testable.
📖
Self-describing
Skills must declare when to use them and when NOT to — so the AI chooses the right tool for the job.
Skills
What to Include — and Avoid
✅ Include
  • Trigger conditions ("use when…", "do NOT use when…")
  • Step-by-step procedure in plain language
  • Expected inputs and outputs
  • Tool calls with rationale for each
  • Failure modes and how to handle them
  • Which agents or contexts this skill is valid in
  • Short examples of before/after state
❌ Avoid
  • Global tone / style rules (→ Rules file)
  • Role definitions (→ Agent)
  • Hardcoding environment-specific paths or URLs
  • Side effects beyond the skill's declared scope
  • Duplicating steps already handled by calling agents
  • Mixing unrelated actions into one skill
  • State that persists beyond the invocation
Skills · Example
Anatomy of a Good Skill
---
name: commit
description: Creates a conventional commit with a co-author footer.
             Use when the user says /commit or asks to create a commit.
             Do NOT use when no staged changes exist, or user just wants to stage files.
---

# Steps
1. Run `git status` — list staged and unstaged changes.
2. Run `git diff --staged` — read the full diff.
3. Run `git log -5` — match the repo's commit message style.
4. Draft a message: <type>: <summary> (under 72 chars).
5. Commit using a HEREDOC to preserve formatting.
6. Append Co-Authored-By footer.
7. Run `git status` to confirm success.

# Never
- Use --no-verify or skip hooks.
- Amend existing commits unless user explicitly asks.
